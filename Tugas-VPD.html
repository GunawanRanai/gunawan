Penerapan Metode Virtual Private Database Atau Row-Level Security Pada Database

  • Gunawan
  • Nov 20, 2022

Pendahuluan

Adanya ancaman terhadap hilang dan rusaknya data yang disimpan, juga kemungkinan akan permasalahan data yang dicuri, maka keamanan pada database diperlukan untuk melindungi data yang telah disimpan agar terhindar dari risiko yang ada.

Pada database bisa menerapkan kontrol akses secara selektif berdasarkan tingkat dari informasi yang dimiliki. Adanya penerapan kontrol akses berdasarkan otoritas yang dimiliki, bisa meningkatkan keamanan yang dimiliki oleh database, dan menjamin kerahasian dari data yang dimiliki.

Virtual Private Database (VPD) merupakan metode yang bertujuan untuk mengatur kontrol akses yang dimiliki hingga ke level baris dan kolom yang sesuai dengan Security Policy yang diterapkan masing - masing teknologi Database Management System (DBMS).

Table pada database yang menerapkan label keamanan pada kolom yang di dalamnya menyimpan label keamanan yang didefinisikan sesuai dengan hierarki skema keamanan yang diterapkan, sehingga keetika user meminta akses ke sebuah row, mekaniskme keamanan akan membandingkan informasi kemanan user dengan informasi kemanan pada row.

Pada Oracle, metode VPD ini berikan nama Oracle Label Security yang mengimplementasikan kontrol akses secara multilevel bersadarkan klasifikasi dari data dan label akses yang dimiliki pengguna aplikasi.

Row-Level Security (RLS) adalah metode fine-grained access control, yang memberikan otorisasi kepada user untuk menampilkan data yang dibutuhkan atau yang diberikan izin, bahkan di dalam tabel. Penerapan atau perintah yang digunakan untuk mengaktifkan security policies berbeda tergantung dari DBMS yang digunakan.

Banyak aplikasi yang membutuhkan RLS pada database relasional, jadi akses user secara individual bisa dibatasi seusai dengan spesifikasi dari row yang di inginkan untuk dibuat security control mandatory. Dengan Mandatory Access Control ini user, pembuat aplikasi, dan administrator database tidak dapat bypass mekanisme RLS tersebut.

Untuk membandingan penerapan metode Row Level Security ataupun Virtual Private Database, pada tulisan ini saya akan melakukan implementasi metode tersebut pada dua DBMS, yaitu MySQL yang tidak memiliki fitur RLS dan PostgreSQL yang memiliki fitur RLS.

Implementasi pada DBMS

Implementasi pada MySQL

Pada MySQL fitur VPD atau sejenisnya tidak dimiliki oleh software ini. Namun pada tulisan ini kita akan menerapkan konsep Row Level Security (RLS) pada DBMS MySQL. Karena pada MySQL sendiri tidak menerapkan Security Policies pada level row, hanya ada untuk passowrd, akses kontrol, manajemen akun, koneksi dienkripsi dan komponen keamanan dan plugin serta data masing. Namun pada kesempatan ini kita akan memanfaatkan Akses kontrol dan manajemen akun untuk membuat Row Level Security.

Untuk menerapkan metode ini, kita menanfaatkan perintah VIEW untuk menampilkan sesuai dengan kebutuhan user. Namun kekurangannya kita harus membuat user sesuai dengan kategori yang ingin kita buat.

Pada implementasi kali ini, kita telah memiliki database karyawan dengan table data_karyawan yang berisi field : id, name, email, region, numberrage, country. Pada kesempatan ini yang akan menjadi acuan adalah country, dimana kita menerapkan konsep row-level security untuk membagi data sesuai dengan wilayah yang ditempati atau di isi

Implemetasi Pada DMBS PostgreSQL

PostgreSQL merupakan relasional database yang mampu mendukung perintah SQL, yang dikembangkan oleh University of California. Pada PostgreSQL sendiri memiliki fitur Row Level Security yang digunakan untuk salah satunya kerahasian dari isi data.

Pada PostgreSQL sendiri memiliki perintah perintah untuk mengaktifkan fitur tersebut dengan Contoh :

ALTER TABLE (nama_database) ENABLE ROW LEVEL SECURITY;

Pada perintah diatas digunakan untuk mengaktifkan fitur Row Level Security pada PostgreSQL.Selanjutnya bisa dengan membuat Policy sesuai dengan kebutuhan, seperti contoh pada perintah berikut.

CREATE POLICY admin_all ON data_customers TO admin USING (true) WITH CHECK (true);

Policy diatas secara implisit menerapkan klausa WITH CHECK secara identic dengan klausa USING.

CREATE POLICY all_view ON data_customers FOR SELECT USING (current_user=user_id); CREATE POLICY user_mod ON data_customers FOR UPDATE USING (current_user =user_id) WITH CHECK ( current_user = user_id);

Pada Row Security Policies (nama metode VPD pada PostgreSQL) bisa lebih spesifik pada perintah, atau role, dan juga keduanya. Policy tersebut bisa diterapkan kepada semua perintah, atau ke SELECT, INSERT, UPDATE, atau DELETE. Beberapa role bisa diterapkan Policy yang diberikan, dan keanggotaan role normal dan aturan perawrisan yang berlaku. Untuk secara lengkapnya bisa dilihat pada percobaan berikut.

Pada langkah ini, kita diharuskan membuat role yang dibutuhkan, dimana role admin sebagai administrator, dan role user_1 serta user_4 sebagai user biasa.

Pada gambar 9 dan 10 yang telah diterapkan policy, dimana role admin bisa melihat semua data dan user_1 hanya bisa melihat data yang kondisi field user_id sama dengan user yang dipakai, sehingga data yang ditampilkan sesuai dengan role yang dipakai.

2.3. Perbedaan Performance Penerapan Role Level Security pada DBMS PostgreSQL

Pada kesempatan ini saya akan membandingan performance pada tiap user yang memiliki jumlah row dan policy berbeda yaitu role admin yang memiliki semua data, user_1 dan user_4 yang bisa menampilkan data sesuai policy.

Pada sub pembahasan ini akan membahas performance dalam hal ini proses untuk menyajikan data. Lama proses pada penyajian data di database, tergantung dari banyaknya data yang dimiliki oleh database tersebut. Pada kesempatan ini kita akan membandingan proses penyajian data dari user admin, serta user_1 dan user_4.

Pada percobaan ini untuk membandingkan performance pada query untuk mencari data, maka akan digunakan kondisi dengan user_id sesuai dengan user yang ada. Berikut kondisi yang akan diterapkan pada percobaan, sebagai berikut.

1. Melakukan query pada role admin dengan 3 query, yaitu melihat semua data yang memiliki data 57000 row, melihat data dengan user_id adalah user_1 yang memiliki data 285 row dan melihat data dengan user_id adalah user_4 yang memiliki data 342.

Pada gambar diatas, menyajikan data waktu yang dibutuhkan database untuk menyajikan data sesuai dengan banyak row yang dimiliki sesuai dengan kondisi. Yaitu pada role admin untuk melihat semua data membutuhkan waktu 139.800 ms, dengan kondisi user_id adalah user_1 membutuhkan waktu 3.456 ms, dan dengan kondisi user_id adalah user_4 dengan waktu 4.720 ms.

2. Melakukan query pada role user_1 untuk melihat semua data yang dimiliki user_1 sebanyak 285 data sesuai dengan policies.

Dalam hal ini waktu yang dibutuhkan role user_1 untuk melakukan query-nya adalah 8.728 ms.

3. Melakukan query pada role user_4 untuk melihat semua data yang dimiliki user_4 sebanyak 342 data sesuai dengan policies.

Dalam hal ini waktu yang dibutuhkan role user_4 untuk melakukan query-nya adalah 7.404 ms.

Dari percobaan diatas membuktikan bahwa, dengan mengingkatkan keamanan pada database, yiatu dengan cara mengaktifkan RLS maka waktu yang dibutuhkan Database untuk menjalankan query tersebut memerlukan waktu yang lebih lama.

Kesimpulan

Perbedaan yang dimiliki antara MySQL dengan PostgreSQL pada penerapan metode Virtual Private Database (VPD) atau Row Level Security (RLS) seperti berikut.

Pada MySQL tidak memiliki fitur RLS namun bisa digunakan metodenya dengan perintah VIEW dan menghasilkan tabel baru, lalu membuat user dan memberikan aksesnya. Hal ini tidak disarankan karena memerlukan waktu lebih lama untuk mengaturnya.

Pada PostgreSQL yang memiliki fitur RLS ini menggunakan perintah ENABLE ROW LEVEL SECURITY sehingga tidak memerlukan untuk membuat user baru, hanya mengatur Policies yang diperlukan dan hak akses yang diizinkan.

Dengan meningkatkan keamanan yang dimiliki agar tidak terjadi bocornya data atau kesalahan , juga akan menurunkan kualitas dari user friendly dari teknologi tersebut, dikarenakan membutuhkan waktu yang lebih untuk menjalankan keamanan yang diterapkan.
